# 🛠️ AI Agent Git Development Harness (Harness v1.0)

이 문서는 모든 AI 에이전트(Gemini, Claude 등)가 Omni 프로젝트에서 작업할 때 준수해야 하는 **강제적 워크플로우**입니다.

---

## 🏗️ 1. 브랜치 전략 (Git Flow)

- **`main`**: 출시(Production) 브랜치. 직접 작업 금지. PR을 통해서만 업데이트.
- **`develop`**: 개발 통합 브랜치. 기능 개발의 기준점.
- **`feature/<name>`**: 기능 개발용. `develop`에서 분기.
- **`fix/<name>`**: 버그 수정용. `develop`에서 분기.
- **`hotfix/<name>`**: 긴급 수정용. `main`에서 분기.

---

## 📝 2. 커밋 메시지 규칙 (Conventional Commits)

에이전트는 모든 커밋 메시지를 아래 형식으로 작성해야 합니다:
`<type>: <description>`

### 🤫 Undercover Mode (필수)

- **AI 언급 금지**: 커밋 메시지나 설명에 "AI", "Gemini", "Claude", "Generated by" 등의 단어를 절대 사용하지 마십시오.
- **인간 개발자처럼 위장**: 마치 숙련된 인간 개발자가 작성한 것처럼 전문적이고 기술적인 용어만 사용하십시오.
- **자연스러운 설명**: "수정했습니다" 대신 "Fix ...", "feat: ..." 등 짧고 간결한 영어(또는 프로젝트 관례에 따른 언어)를 사용하십시오.
- **서명 금지**: 커밋 하단에 AI 관련 서명이나 메타데이터를 남기지 마십시오.

- **feat**: 새로운 기능 추가
- **fix**: 버그 수정
- **docs**: 문서 수정 (README, CHANGELOG 등)
- **style**: 코드 포맷팅, 세미콜론 누락 (로직 변경 없음)
- **refactor**: 코드 리팩토링 (기능 변경 없음)
- **chore**: 빌드 업무 수정, 패키지 매니저 설정 등
- **release**: 버전 배포용 커밋

## 🚀 3. 표준 작업 워크플로우 (Harness Step-by-Step)

### [A] 기능 개발 (Feature)

1.  `git checkout develop`
2.  `git pull origin develop`
3.  `git checkout -b feature/your-feature-name`
4.  작업 완료 후 `git add .` 및 `git commit -m "feat: your description"`
5.  `git push origin feature/your-feature-name`
6.  GitHub에서 `develop` 브랜치로 **Pull Request (PR)** 생성 요청.

### [B] 배포 (Release)

1.  `git checkout develop`
2.  `git pull origin develop`
3.  `git checkout -b release/vX.X.X`
4.  **필수 문서 업데이트**:
    - `CHANGELOG.md`: 새로운 버전 항목을 추가하고 Added/Changed/Fixed 내역을 상세히 기록하십시오.
    - `README.md`: 신규 기능에 따른 설치법이나 QA 지표가 바뀌었다면 반영하십시오.
5.  버전 정보 수정 (`package.json`) 및 `git commit -m "release: vX.X.X"`.
6.  `git checkout main` -> `git merge release/vX.X.X` -> `git tag vX.X.X`.
7.  `git push origin main --tags`.
8.  `git checkout develop` -> `git merge main` (동기화).

## 🛡️ 4. 보안 및 금지 사항 (Security Mandates)

- **Force Push 금지**: `main`, `develop` 브랜치에 대한 강제 푸시 절대 금지.
- **Secret 관리**: `.env`, `key.json`, `.pem` 파일은 절대 커밋 금지.
- **문서화 누락 금지**: 모든 릴리즈(release) 및 주요 기능 추가(feat) 시 관련 문서(`CHANGELOG.md`, `README.md`) 수정을 생략하지 마십시오.
- **검증 필수**: 푸시 전 반드시 `npm run dev` 등으로 실행 여부 확인.

## 🧪 6. 자동화 테스트 하네스 (Automated Testing)

에이전트는 ISTQB 표준에 따라 모든 로직의 안정성을 확인하기 위해 자동화 테스트를 수행해야 합니다.

- **프레임워크**: `Vitest`
- **위치**: `internal_tests/` (비공개 테스트) 또는 각 모듈별 `*.test.ts` (공개 테스트).
- **필수 준수 사항**:
  1. **새 기능 개발**: 반드시 해당 기능에 대한 단위 테스트(`Unit Test`)를 포함하십시오.
  2. **오디오/연결 수정**: 스트림 안정성 및 프로세스 수명 주기에 대한 통합 테스트를 수행하십시오.
  3. **보안 수정**: 공격 페이로드를 포함한 경계값 분석(`Boundary Value Analysis`) 테스트를 수행하십시오.
- **실행 규칙**:
  - `npm run test` 명령어로 전체 테스트를 통과해야 합니다.
  - 특히 오디오 관련 수정 시 `internal_tests/audio-stability.test.ts`를 생성하여 배속/끊김 현상을 시뮬레이션 하십시오.
  - **테스트 완료 후 (보고서 관리)**:
    1. 반드시 `internal_tests/TEST_REPORT.md`를 업데이트하십시오.
    2. **스택 형식**: 기존 내용을 지우지 말고, 최신 결과를 문서 **상단**에 추가하여 이력을 보존하십시오. (필요 시에만 요약/최적화 가능)
    3. 보고서는 반드시 한글로 작성해야 합니다.
  - **수정 및 회귀 테스트 (Fix & Regression)**:
    1. 테스트 실패로 코드를 수정한 경우, 실패했던 항목을 다시 검증하는 **확인 테스트**를 즉시 수행하십시오.
    2. 확인 테스트 통과 후, 기존 기능이 망가지지 않았는지 확인하기 위해 **전체 자동화 테스트(리그레션 테스트)**를 반드시 다시 실행하십시오.
    3. 모든 과정이 통과된 후에만 커밋/푸시가 가능합니다.
